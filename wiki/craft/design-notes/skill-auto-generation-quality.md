# 设计笔记：Skill 自动生成为什么需要两道质量关卡？

## 设计选择

MemOS 的 Skill 自动生成流程有两道质量关卡：
1. **LLM 决策**时的 confidence 阈值（≥ 0.7 升级已有 Skill，< 0.3 新建 Skill）
2. **最终写入**时的质量评分（≥ 6 分才写入数据库）

## 直觉反应

"一道关卡不够吗？要么设高一点 confidence 阈值，要么只看质量评分，两道都要是不是多余？"

## 设计考量

### 两道关卡回答不同的问题

Confidence 阈值回答的是：**"这段经验跟已有 Skill 是什么关系？"**

- ≥ 0.7：高度相关 → 大概率是对已有 Skill 的补充，执行 UPDATE
- < 0.3：几乎无关 → 大概率是全新经验，执行 CREATE
- 0.3 ~ 0.7：灰色地带 → 跳过，等更多证据积累

质量评分回答的是：**"这个 Skill 值不值得保留？"**

一个 Skill 可能是全新的（confidence < 0.3 → CREATE），但内容太泛（"做事要仔细"）或太碎（"第 37 行加个逗号"），这种 Skill 即使新颖也没有复用价值。质量评分过滤的是这类"新但没用"的 Skill。

### 没有 confidence 阈值会怎样？

假设只有质量评分：

Agent 在 10 次不同对话中遇到了类似的 pytest 问题，每次都提炼出一个 Skill。10 个 Skill 的质量都够 6 分，于是全部写入——但它们本质上是同一条经验的 10 种措辞。

结果：Skill 库膨胀，检索时 Agent 被 10 个相似 Skill 淹没。Auto-recall 注入时，System Prompt 里塞了 10 段大同小异的内容，上下文被浪费。

Confidence 阈值的作用是**在写入之前先确定"这是升级还是新建"**，避免同一经验反复新建。

### 没有质量评分会怎样？

假设只有 confidence 阈值：

Agent 在一次调试中学到了"这个 bug 是因为我少打了一个括号"。Confidence < 0.3（确实是新经验），直接写入。但这个 Skill 对其他任务毫无参考价值——它太具体、太偶然、没有泛化能力。

结果：Skill 库充斥着低价值的"一次性教训"，信噪比下降。

质量评分的作用是**保证只有真正有复用价值的经验才被沉淀**。评分标准通常包括：

- **泛化能力**：能否应用于类似场景？
- **具体程度**：是否包含可操作的步骤？
- **独特性**：是否提供了非显而易见的洞察？

### 灰色地带的处理方式很关键

Confidence 在 0.3 ~ 0.7 之间的经验直接跳过，这看起来会丢信息。但这是一个有意为之的设计：

- 如果一段经验确实重要，Agent 会在后续任务中多次遇到它
- 多次遇到 → 更多上下文积累 → 下次 confidence 更高或更低，脱离灰色地带
- 不重要的经验自然不会反复出现 → 自然淘汰

这是一种**"让时间说话"的策略**——不急于在不确定时做判断，而是等更多证据。

## 对比视角：Hermes 的手动策展

Hermes 的 Skill 创建完全由 Agent 主动触发，没有自动质量关卡。触发条件是启发式规则：

- 5+ 工具调用
- 错误恢复
- 用户纠正
- 非显而易见的工作流

这意味着质量完全取决于 Agent 的判断力。好处是零额外开销（不需要 LLM 评分），坏处是社区反馈 Skill 重复率高——Agent 没有全局视野，不知道自己创建的 Skill 和别人的重复了。

| 维度 | MemOS（两道关卡） | Hermes（Agent 判断） |
|------|----------------|-------------------|
| 去重能力 | 强（confidence 阈值） | 弱（无全局检索） |
| 质量下限 | 高（6 分门槛） | 取决于 Agent | 
| 额外成本 | 有（LLM 评分调用） | 无 |
| 维护负担 | 低（自动维护） | 中（需要人工清理） |

## 启发

**质量控制的两个维度——"是否重复"和"是否有价值"——不能用同一个机制解决。** 去重是关系判断（和已有知识的关系），质量是绝对判断（这条知识本身好不好）。混在一起处理，要么漏掉重复，要么误杀好内容。

这个模式在很多知识管理场景中都适用：代码审查（这个改动是重构还是新功能？改动本身质量如何？）、文档管理（这篇文档和已有文档重复吗？文档本身质量如何？）、甚至投资研究（这个信息是已知事实的重复表述吗？信息本身对投资决策有价值吗？）。
