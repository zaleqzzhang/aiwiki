# 设计笔记：OpenClaw 上下文压缩算法（Compaction）

## 设计选择

OpenClaw 在上下文窗口快满时，使用**分块摘要**策略压缩历史对话——把旧消息分成块，逐块让模型生成摘要，用摘要替换原文。源码位于 `src/agents/compaction.ts`。

## 直觉反应

"让模型总结自己的对话历史？这不就是用 AI 压缩 AI 的输入吗——压缩过程本身也要消耗 token，而且摘要质量取决于模型的总结能力。会不会丢失关键信息？"

## 设计考量

### 为什么不直接截断（丢弃最旧的消息）？

截断是最简单的策略，但问题明显：
- 旧消息里可能包含**关键决策**（"我们选了方案 A 因为..."）
- 任务开头的**目标定义**会被丢掉，模型"忘了"自己在干什么
- **不透明标识符**（UUID、文件哈希）一旦丢失就无法恢复

摘要可以选择性保留关键信息，丢弃细节。代价是压缩过程本身消耗 token 和时间。

### 分块参数的工程权衡

OpenClaw 的分块参数不是拍脑袋的——每个数字背后都有考量：

| 参数 | 值 | 设计意图 |
|------|-----|---------|
| `BASE_CHUNK_RATIO` | **0.4**（每块占上下文 40%） | 够大才能让摘要模型看到足够上下文，生成连贯摘要 |
| `MIN_CHUNK_RATIO` | **0.15**（每块最小 15%） | 防止块太小导致摘要碎片化、丢失跨消息的关联 |
| `SAFETY_MARGIN` | **1.2**（20% 缓冲） | 压缩后留出余量给后续对话，避免压缩完马上又触发压缩 |
| `SUMMARIZATION_OVERHEAD_TOKENS` | **4096** | 摘要指令本身占的 token，预留出来避免压缩过程中超限 |

**40% 的 chunk ratio 是核心决策**：太小（比如 10%）→ 摘要碎片化，每块看不到足够上下文；太大（比如 80%）→ 块数太少，压缩粒度粗，可能一次丢掉大量有用信息。40% 意味着 200K 上下文窗口大约切 2-3 块，每块 80K token 左右——这个粒度既能保持连贯性，又能实现有意义的压缩。

### 三级摘要的防御式设计

```
summarizeInStages()    ← 顶层调度：判断消息量，决定分几块
    ↓
summarizeChunks()      ← 分块处理：每块独立摘要，最多重试 3 次
    ↓
summarizeWithFallback() ← 兜底策略：排除过大消息后重试
    ↓                     最终兜底：返回 "No prior history."
```

**为什么需要三级？** 因为压缩操作本身可能失败：
- 模型可能拒绝（输出太长、格式不对）
- 某条消息可能异常大（一整个文件的内容）导致超限
- 网络/API 可能超时

三级策略确保：正常情况走第一级（高质量分块摘要）→ 异常时走第二级（重试+排除问题消息）→ 极端情况走第三级（彻底放弃历史，返回空白）。**宁可丢失所有历史也不要让 Agent 崩溃。**

### 保留项的选择

压缩不是无脑缩减——有些信息**绝对不能丢**：

- **活跃任务**：正在做什么（丢了就不知道目标了）
- **重要决策**：为什么选了方案 A（丢了可能重新做错误决策）
- **TODO 列表**：还有什么没做（丢了就遗漏了）
- **不透明标识符**：UUID、文件哈希等必须原文保留（摘要后面目全非）
- **用户承诺**："我答应你明天做 X"（丢了就失信了）

这个保留列表本身就是 Context Engineering 的精髓——**知道什么不该丢，比知道什么该留更重要。**

### 超时保护：5 分钟硬上限

`EMBEDDED_COMPACTION_TIMEOUT_MS = 300000`（5 分钟）

为什么需要超时？因为压缩是用模型做摘要——如果上下文巨大，摘要过程本身可能要几分钟。加上可能的重试，没有超时保护的话，用户可能等很久。5 分钟是"在用户耐心耗尽前必须完成"的工程判断。

还有写锁（防止两个压缩操作并发执行，导致数据混乱）和可配置的压缩模型（可以用更便宜/更快的模型做摘要，不一定要用主对话模型）。

## 对比视角

### vs Hermes 的策略

Hermes 采用的是有界热记忆——对话长度有硬限制，超了就截断旧消息。简单粗暴但可预测。

OpenClaw 的分块摘要更复杂但更"聪明"——试图在压缩中保留最关键的信息。代价是实现复杂度更高、压缩过程本身消耗资源。

| 维度 | Hermes（截断） | OpenClaw（分块摘要） |
|------|---------------|-------------------|
| 实现复杂度 | 低 | 高 |
| 信息保留质量 | 低（直接丢弃） | 中（摘要保留要点） |
| Token 开销 | 零 | 压缩过程消耗 token |
| 可预测性 | 高 | 中（依赖摘要质量） |
| 延迟 | 无 | 有（最长 5 分钟） |

### vs 精细化修剪（Pruning）

OpenClaw 还有一个互补策略：Pruning（修剪），处理的是单条消息的裁剪——头尾保留、中间省略，裁剪不超过 50%。

Compaction 处理**历史维度**的压缩（旧对话 → 摘要），Pruning 处理**单条消息维度**的裁剪（超长工具输出 → 头尾保留）。两者配合，从两个维度控制上下文膨胀。

## 启发

1. **Context Engineering 不只是"选择放什么进去"，还包括"怎么优雅地丢弃"**。压缩算法是 Context Engineering 从理论到工程的关键转化。

2. **防御式设计在 Agent 系统中至关重要**。三级摘要 + 超时保护 + 写锁——每一层都是为了应对"正常情况下不会发生但一旦发生就很严重"的场景。Agent 系统比传统软件更需要防御式设计，因为模型的行为本质上不可完全预测。

3. **40% chunk ratio 是经验值，不是理论最优**。不同的上下文窗口大小、不同的任务类型，最佳 chunk ratio 可能不同。这是一个值得持续调优的参数。

4. **"宁可丢失历史也不要崩溃"**是正确的兜底哲学。`"No prior history."` 看起来很暴力，但对用户来说，一个"忘了之前做什么但还能继续工作"的 Agent 远好于一个"完全卡死"的 Agent。
