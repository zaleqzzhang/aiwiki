# 设计笔记：OpenClaw 时间衰减记忆

## 设计选择

OpenClaw 的记忆召回使用**时间衰减函数**对记忆条目加权：

```
衰减系数 = e^(-λ × 天数)
其中 λ = ln(2) / 半衰期（默认 30 天）
```

效果：30 天减半，60 天剩 1/4，90 天剩 1/8。

## 直觉反应

第一反应是"这很合理"——最近的记忆更可能与当前任务相关。人类记忆也有类似的衰减曲线（Ebbinghaus 遗忘曲线）。

但仔细想，**不是所有记忆都应该衰减**。用户的项目偏好（"我喜欢 TypeScript"）、架构决策（"这个项目用 monorepo"）是不应该随时间衰减的。纯粹按时间衰减，意味着 90 天前确定的架构决策只有 1/8 的权重——这显然不对。

## 设计考量

### 角度一：为什么用指数衰减而非线性衰减？

指数衰减的好处是**近期变化快、远期变化慢**。前 30 天从 100% 降到 50%，但从 50% 到 25% 又花了 30 天。这意味着近期记忆的区分度很高（昨天 vs 上周的差异明显），远期记忆趋于平等（3 个月前 vs 4 个月前差别不大）。

线性衰减的问题是"悬崖"——到了某个时间点突然降到 0。指数衰减渐近于 0 但永不到达——理论上任何记忆都有被召回的可能。

### 角度二：半衰期 30 天是否合适？

这个参数选择暗含一个假设：**Agent 的工作节奏是"月度项目"尺度**。如果你的 Agent 做的是长期项目（3-6 个月），30 天半衰期太短。如果做的是日常零碎任务，30 天又太长。

对比 Hermes 的方案：3,575 字符硬上限 + 全文搜索（FTS5）。Hermes 不做时间衰减——所有记忆等权，超过上限时由 Agent 自己决定保留什么。这是截然不同的哲学：**系统策展 vs Agent 策展**。

### 角度三：与 BM25 + 向量召回的交互

OpenClaw 的完整召回流程是：BM25 + 向量双路召回 → 时间衰减加权 → 排序取 Top-N。时间衰减是在相关性评分之后叠加的一层权重。

这意味着即使一条旧记忆的 BM25/向量相关性分数很高，时间衰减也会把它压低。一个可能的改进是：对不同类型的记忆使用不同的衰减参数——事实性记忆（偏好、架构决策）衰减慢或不衰减，事件性记忆（具体操作、调试过程）正常衰减。

### 角度四：MEMORY.md vs 每日笔记的隐式分层

OpenClaw 实际上通过双层记忆结构部分缓解了衰减问题：
- **MEMORY.md**（长期记忆）：注入系统提示词，截断到 200 行，不做时间衰减——这是"永不遗忘"的层
- **memory/日期.md**（每日笔记）：走 BM25 + 向量 + 时间衰减召回——这是"逐渐淡忘"的层

所以时间衰减只作用于每日笔记层，长期记忆通过直接注入绕过了衰减。这是一个聪明的分层设计——但前提是用户或 Agent 要主动把重要信息从每日笔记"升格"到 MEMORY.md。

## 对比视角

| 系统 | 记忆淘汰策略 | 哲学 |
|------|------------|------|
| OpenClaw | 时间衰减 + MEMORY.md 分层 | 系统自动衰减，长期记忆手动维护 |
| Hermes | 硬上限 + Agent 自选 | Agent 自治，系统不干预 |
| MemOS | 三层去重（哈希→向量→LLM） | 先全量捕获，再智能去噪 |
| LangSmith Agent Builder | Semantic Memory + Episodic Memory | 类型分离，不同类型不同策略 |

## 启发

时间衰减是一个"足够好"的默认策略，但它的隐含假设（"新的更重要"）不总是成立。如果你在设计记忆系统，考虑：

1. **分层比单一衰减更重要**——区分"需要永远记住的事实"和"逐渐淡忘的事件"
2. **衰减参数应该可配置**——不同场景（长期项目 vs 日常任务）需要不同的半衰期
3. **类型感知衰减**——偏好/决策不衰减，操作/调试正常衰减
