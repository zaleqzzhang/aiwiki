# 设计笔记：Hermes 为什么选择 3,575 字符的记忆上限？

## 设计选择

Hermes 的 Prompt Memory（始终注入到每次对话的记忆）限制在 3,575 字符：MEMORY.md 占 2,200 字符，USER.md 占 1,375 字符。

## 直觉反应

"太小了吧？一篇短文章都放不下。为什么不多给点？35K 字符？350K？"

## 设计考量

### 约束即特性

有限空间 → 强制策展 → **只保留最重要的信息**。

如果给无限空间，Agent 什么都存，最后什么都找不到——或者更准确地说，什么都注入到 Prompt 里，上下文被稀释，模型注意力分散。这和人的工作记忆类似：**不是容量越大越好，而是"知道什么该放进来"比"放得多"更重要**。

Hermes 的 Periodic Nudge 机制强化了这个设计：会话中间自动触发，Agent 回顾并策展记忆，主动决定保留什么。这不是 bug（空间不够用），而是 feature（强制 Agent 学会取舍）。

### Token 经济学

Prompt Memory 在**每次 API 调用**都会注入——不是每次对话，是每次 API 调用。

3,575 字符 ≈ ~1,200 tokens。如果一次对话有 20 轮 API 调用，仅记忆注入就消耗 24,000 tokens。

如果扩大到 10K 字符 ≈ ~3,500 tokens → 70,000 tokens/对话仅用于记忆注入。35K 字符？那就是 ~240,000 tokens/对话——可能超过模型的上下文窗口。

这是硬约束，不是审美选择。

### 分层替代扩容

不够的部分不是放弃，而是**推到更低层**：

```
常用 → Prompt Memory（始终在线，~1,200 tokens）
偶尔用 → Session Search（按需检索，FTS5）
程序性知识 → Skills（任务触发时加载）
用户特性 → Honcho（长期个人助理场景，12 个身份层）
```

这是用**架构解决容量问题**，而不是简单加大缓冲区。每一层的检索成本和注入策略不同，形成了一个"越常用越靠前、越偶尔越靠后"的分层结构。

### 压缩而非截断

当对话超出上下文限制时，Hermes 不是简单截断，而是：

1. 辅助模型扫描全量对话
2. 提取有价值内容写入 MEMORY.md（3,575 字符内）
3. Lineage（参考链）保留在 SQLite，可追溯原始内容

这意味着即使记忆空间有限，**原始数据始终完整保留**——只是访问路径不同。热数据在 Prompt Memory，温数据在 Session Search，冷数据在 SQLite。

## 对比视角：MemOS 选择了另一条路

MemOS 的选择截然不同：无界记忆（35,000+ 条）+ 智能去重。

这不意味着 MemOS "更好"。它的代价是：
- API 成本 $200-300/月 vs Hermes $10-50/月
- 检索复杂度更高（BM25 + 向量 + RRF 融合）
- 需要更多工程投入维护记忆质量

两种选择反映了不同的产品定位：Hermes 是个人助手（快速启动、低成本），MemOS 是团队协作系统（长期运行、高投入高回报）。

## 启发

**"带着约束做设计"经常比"给足资源"产出更好的架构。**

这和微服务中 bounded context 的逻辑一致：不是因为一个服务"装不下"才拆分，而是拆分本身带来了更清晰的职责边界。Hermes 的 3,575 字符限制迫使它发明了四层记忆架构——如果一开始就给无限空间，可能就不会有这个精巧的分层设计。

写文章时可以引申：**在 Agent 系统设计中，约束往往是创新的催化剂**。
