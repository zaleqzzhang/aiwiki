# 设计笔记：MemOS 为什么需要三层去重？

## 设计选择

MemOS 对 Agent 记忆采用三层去重机制：内容哈希精确匹配 → Top-5 向量相似度检测（阈值 0.75）→ LLM 最终裁决（DUPLICATE / UPDATE / NEW）。

## 直觉反应

"一层哈希去重不就够了？为什么还要搞向量检索和 LLM 裁决，不是过度工程吗？"

## 设计考量

### 语义重复 ≠ 字面重复

哈希去重只能处理完全相同的记忆。但 Agent 经常用不同措辞表达相同的经验：

- 记忆 A："用户偏好简短回复，不超过两段"
- 记忆 B："回复要精简，控制在 2 段以内"

这两条记忆传递的信息完全一致，但哈希值不同。如果只做哈希去重，35,000 条记忆里可能 30% 是语义重复——Agent 检索时会被同义表述淹没，真正有用的信息反而被稀释。

向量相似度检测的引入正是为了捕获这种"意思一样但说法不同"的重复。阈值 0.75 是一个经验值：太低（0.5）会误伤相关但不重复的记忆，太高（0.9）会漏掉措辞差异较大的重复。

### LLM 裁决处理灰色地带

有些情况连向量相似度也无法判定。例如：

- 记忆 A："Python 项目用 pytest 做单元测试"
- 记忆 B："Python 项目用 pytest 做单元测试，覆盖率目标 80%"

B 是 A 的扩展版本。它们不是重复（B 包含新信息），但也不完全独立。正确的操作是 **UPDATE**——用 B 替代 A，保留新信息，避免冗余。

这种"部分重叠 + 部分新增"的判断需要语义理解，不是相似度分数能解决的。LLM 裁决在这里充当最终仲裁者，输出三个判定之一：

- **DUPLICATE**：完全冗余，丢弃新记忆
- **UPDATE**：有新信息，合并到旧记忆
- **NEW**：独立知识，新建记忆

### 三层是成本梯度设计

```
第一层：哈希（成本 ≈ 0）  → 过滤 ~20% 完全重复
第二层：向量（成本 ≈ 低）  → 过滤 ~15% 语义重复
第三层：LLM（成本 ≈ 高）   → 处理 ~5% 灰色地带
                           → 剩余 ~60% 为真正的新知识
```

如果所有记忆都送 LLM 裁决，成本会爆炸（35,000 条 × LLM 调用）。三层漏斗的设计是**用便宜的方法先过滤大部分，只把难判断的留给贵的方法**。这和搜索引擎的"粗排 → 精排 → 重排"思路一致。

### 幂等性保障

三层去重还解决了一个工程问题：**Agent 重启或 Hook 重复触发时的幂等性**。

如果没有去重机制，`agent_end` Hook 每次触发都会写入新记忆——Agent 重启 10 次，同一段对话就被存了 10 遍。哈希层保证了精确重复不会入库，向量层保证了措辞微调的重复也会被识别。

## 对比视角：Hermes 选择了"少存"而非"去重"

Hermes 根本不需要三层去重——因为它从源头控制了记忆量。3,575 字符的硬限制 + Agent 自主策展 = 天然不会膨胀。

两种策略的哲学差异：

| 维度 | MemOS | Hermes |
|------|-------|--------|
| 策略 | 先全量存，再智能去重 | 先精选存，不需要去重 |
| 成本 | 去重机制本身有成本 | 策展由 Agent 承担（token 成本） |
| 风险 | 去重失败 → 记忆膨胀 | 策展失误 → 丢失重要信息 |
| 适用规模 | 团队级（35,000+ 条） | 个人级（~100 条有效记忆） |

这不是哪个更好的问题，而是**规模不同，方案不同**。一个人的笔记本不需要搜索引擎，但一个团队的知识库需要。

## 启发

**数据质量问题往往需要多层防线。** 单一机制总有盲区——哈希看不到语义重复，向量算不清部分重叠，LLM 成本太高不能普筛。组合使用，让每一层处理自己擅长的那类问题，是一个通用的工程模式。

这和安全领域的"纵深防御"原则相通：不依赖单一防线，而是让多层机制各守一段。
