# 设计笔记：Hermes 子 Agent 为什么选择同步阻塞？

## 设计选择

Hermes 的 `delegate_task` 采用同步阻塞模式：父 Agent 发起委派后停下来等，直到所有子 Agent 完成才继续。子 Agent 的中间推理过程不返回给父 Agent，只返回最终 summary。

## 直觉反应

"同步阻塞？这不是最原始的并发模式吗？为什么不用异步？子 Agent 跑的时候父 Agent 闲着多浪费。"

## 设计考量

### 上下文污染是更大的问题

异步的好处是父 Agent 可以边等边做别的事。但代价是：子 Agent 的中间状态——思考过程、工具调用输出、错误重试——全部涌入父 Agent 的上下文窗口。

一个子 Agent 处理一个文件搜索任务，中间可能产生 20 条工具调用记录、5 次错误重试、10 段推理过程。如果这些全部回传，父 Agent 的上下文会被无关信息淹没。它在试图处理下一个子任务时，注意力被"子 Agent 3 正在重试 SSH 连接"之类的噪音分散。

同步阻塞 + 只返回 summary 的设计**从源头切断了上下文污染**。父 Agent 看到的是干净的结果，不是混乱的过程。

### Token 效率的量化差距

假设一次复杂任务委派 3 个子 Agent，每个子 Agent 产生 15,000 tokens 的中间输出：

- **异步模式**：父 Agent 上下文累积 3 × 15,000 = 45,000 tokens 的子 Agent 输出。加上父 Agent 自身的上下文，很容易撞到上下文窗口上限。
- **同步阻塞 + summary**：每个子 Agent 只返回 ~500 tokens 的摘要。父 Agent 上下文仅增加 3 × 500 = 1,500 tokens。

30 倍的差距。在 token = 成本 = 延迟的 Agent 经济学里，这不是小事。

### 工具剥夺降低了风险

同步阻塞还有一个前提：子 Agent 被剥夺了大量工具。

- 不能 `delegate_task`（防止递归）
- 不能 `clarify`（无法与用户交互）
- 不能访问 `memory`（保护共享记忆）
- 不能 `execute_code`（避免复杂脚本）

这意味着子 Agent 的行为空间被大幅限制——它只能用剩余工具完成特定任务。行为空间小 → 可预测性高 → 不需要父 Agent 实时监控。同步等一个确定性高的结果，比异步监控一个不可控的过程，工程上更可靠。

### 深度限制是安全网

`MAX_DEPTH = 2` 意味着子 Agent 不能再委派子子 Agent。最多两层。

这个限制看起来粗暴，但解决了一个深层问题：**递归委派的不可控性**。如果允许无限嵌套，一个任务可能 spawn 出几十个子 Agent，每个都在消耗 API 调用和 token——而父 Agent 对此毫无感知。

深度限制 + 同步阻塞 = 父 Agent 始终知道"现在有多少个子 Agent 在跑、什么时候会结束"。这是可审计性的基础。

## 对比视角：OpenClaw 选择了异步编排

OpenClaw 的设计截然不同：

- 异步执行（子 Agent 独立运行，不阻塞父 Agent）
- Steer 机制（父 Agent 可以向运行中的子 Agent 发消息调整方向）
- 持久化与恢复（子 Agent 崩溃后可从断点恢复）
- 无深度限制（理论上可以多层嵌套）

OpenClaw 的异步设计解决了 Hermes 的一个痛点：**长任务场景**。如果一个子 Agent 要跑 30 分钟，同步阻塞意味着父 Agent 也傻等 30 分钟。Hermes 目前没有超时机制——子 Agent 可以无限运行，父 Agent 无法中止。

但 OpenClaw 付出的代价是上下文管理的复杂度。异步 + Steer + 多层嵌套 = 状态空间爆炸。调试一个四层嵌套的异步多 Agent 任务，比调试 Hermes 的两层同步委派难一个数量级。

| 维度 | Hermes（同步） | OpenClaw（异步） |
|------|---------------|----------------|
| 上下文污染 | 无（只返回 summary） | 需要主动管理 |
| Token 效率 | 高 | 低（上下文膨胀） |
| 长任务支持 | 差（无超时机制） | 好（Steer + 恢复） |
| 调试复杂度 | 低 | 高 |
| 最大嵌套 | 2 层 | 无限制 |

## 启发

**"简单"和"原始"是两回事。** 同步阻塞看起来原始，但在特定约束下（有限上下文、Token 经济学、安全性要求），它恰好是最合理的设计。

Hermes 的选择告诉我们：**不要在不需要的地方引入复杂性**。异步是一种能力，但也是一种负担。如果你的子 Agent 任务粒度足够小（几分钟内完成）、行为空间足够窄（工具被剥夺）、结果足够确定（只需要 summary），那同步阻塞就是最优解。

反过来，如果你的场景确实需要长时间运行、动态调整、崩溃恢复——那就应该选 OpenClaw 的异步模式。**没有最佳架构，只有最匹配场景的架构。**
