# OpenClaw vs. Hermes Agent：两种 AI Agent 的生存哲学

> 一个想连接一切，一个想越用越聪明。这不只是工具之争——是 AI Agent 行业对"什么才重要"的根本分歧。

## 不是选哪个更好的问题

如果你现在问"该选 OpenClaw 还是 Hermes"，你大概率问错了问题。

这两个项目代表的是两条完全不同的技术路线，回答的是两个不同的问题：**你的 Agent 需要在多少个平台上跑**，还是**你的 Agent 需要越用越好**。

前者是 OpenClaw 的地盘。后者是 Hermes 的赌注。

理解这个分歧，比记住任何一份功能对比表都有价值。

## 先看数据

| 维度 | OpenClaw | Hermes |
|------|----------|--------|
| GitHub Stars | 345K+（6 个月） | 64K+（2 个月） |
| 平台集成 | 50+（WhatsApp、Slack、Telegram、Discord……） | 6 个 |
| Skill 生态 | 10,000+（ClawHub 社区市场） | 118 个（自生成 + 内置） |
| 多 Agent 并发 | 8 路（可配置） | 3 路（硬编码） |
| 安全记录 | 4 天 9 个 CVE（含 CVSS 9.9） | **零** Agent 特定 CVE |
| 运行成本 | Token 开销更大（+12%） | $0-150/月，$5 VPS 可跑 |
| 核心哲学 | Agent as a system to orchestrate | Agent as a mind to develop |

数据很清楚：OpenClaw 在规模和连接性上碾压，Hermes 在学习能力和安全性上占优。

但数据看不到的是**为什么**会这样。

## OpenClaw：连接一切，然后再说

OpenClaw 的创始人 Peter Steinberger 把它定位为"Android for AI agents"。这个类比非常精准——Android 的核心价值不是系统本身多优雅，而是**什么设备都能装**。

OpenClaw 继承了这个基因：

**50+ 消息平台集成。** 你的 Agent 可以同时存在于 WhatsApp、Telegram、Discord、Slack、Teams、iMessage——用户在哪，Agent 就在哪。跨渠道持久化是 OpenClaw 病毒式增长的核心引擎：一个人在 Slack 上开始的对话，可以在 Telegram 上继续。

**ClawHub 社区市场。** 10,000+ Skills 覆盖编码、写作、数据分析、DevOps 等几十个领域。长尾需求由社区解决——你想让 Agent 做什么怪事都有人写好了 Skill。

**异步编排。** OpenClaw 的多 Agent 协作采用"交响乐团"模式：父 Agent 定义拓扑，子 Agent 异步执行，用 Steer 消息实时引导方向。你可以在子 Agent 跑着的时候说"换方案 B"——它不会崩溃，会调转方向。

这套设计的核心逻辑是：**先让 Agent 无处不在，再解决其他问题。**

问题是，"其他问题"里包含了一颗定时炸弹。

## 安全：连接广度的结构性代价

2026 年 3 月，OpenClaw 在 4 天内被披露 9 个 CVE，其中包括一个 CVSS 9.9 的严重漏洞。

数据更触目惊心：

- **CVE-2026-25253**（CVSS 8.8）：WebSocket 自动连接暴露认证 Token，一键入侵
- **135,000+ 实例暴露公网**，跨 82 个国家
- **ClawHub 恶意 Skill 率 12-20%**：Koi Security 审计 2,857 个 Skills，发现 341 个恶意，其中 335 个来自同一攻击组织
- 恶意 Skill 分发 **Atomic macOS Stealer**——直接偷密钥

微软建议不在个人或企业工作站运行。Cisco 称 OpenClaw 类个人 AI Agent 是"安全噩梦"。

**这不是 Bug——是架构。** OpenClaw 最初设计为消费级本地工具，信任假设是"本地网络是安全的"。当它成长为网络化 Agent 并拥有 10,000+ 第三方 Skill 时，这些信任假设全部坍塌。

**连接广度和安全深度不可兼得。** 这不是 OpenClaw 的工程能力问题，而是任何开放生态都面临的结构性张力——你连接的越多，攻击面就越大。

## Hermes：先学会一件事，再学下一件

Hermes 选了完全不同的方向。Nous Research 把它定位为"the agent that grows with you"——不是连接更多，而是越用越聪明。

**学习循环。** Hermes 的核心创新是一个四步自我改进闭环：

1. **Periodic Nudge**：会话中途自动触发，Agent 回顾并策展记忆
2. **Skill 自主创建**：5+ 工具调用、错误恢复、用户纠正——任何复杂交互都可能被提炼为 Skill
3. **Skill 自我优化**：不只创建，还能 patch 已有 Skill，越用越精准
4. **Session Search**：FTS5 全文检索，~10ms 从 10,000+ 文档中找到相关经验

效果：使用自创 Skill 的 Agent 完成任务比新实例快 **40%**（Nous Research 数据）。

**四层记忆架构。** 而不是 OpenClaw 的"文件即记忆"扁平模式：

```
Prompt Memory（始终在线，3,575 字符）—— 最常用的核心信息
Session Search（按需检索，FTS5）—— 特定话题触发
Skills（渐进加载）—— 程序性知识
Honcho（用户建模，12 个身份层）—— 长期个性化
```

**约束催生架构。** 3,575 字符的记忆上限看起来很小，但这个约束迫使 Hermes 发明了分层策展——只有最重要的信息留在"始终在线"的位置，其余按需检索。这和无限存储的 MemOS（$200-300/月）形成了有趣的对比：有时候约束是创新的催化剂。

**安全副产品。** Skill 自生成意味着不需要从外部市场下载代码——直接绕过了供应链攻击向量。结果：零 Agent 特定 CVE。

但代价也很明确：6 个平台 vs 50+，冷启动阶段没有优势，知识不跨领域迁移。

## 真正的分歧：时间复利

传统软件中，深度（做精一件事）和广度（覆盖更多场景）是静态权衡——你选了一端就是那一端。

**Agent 领域有一个独特变量：时间复利。**

深度方向有复利效应：运行两个月的 Hermes 实例，积累了关于你的系统、偏好、常见模式的机构知识。这种积累是指数增长的——前 1,000 次编译错误修复比第 1 次快得多。

广度方向是线性增长：多连一个平台就是多一个平台，不会因为连了 Slack 就让 GitHub 集成变好。

**短期，广度赢。** 新装一个 OpenClaw，立即就能在 50+ 平台上用。新装一个 Hermes，跟新装的 OpenClaw 差不多——优势要积累很久才显现。

**长期，深度赢。** 但前提是你愿意等。

## 多 Agent 协作：交响乐团 vs 总承包商

这个分歧在多 Agent 架构上体现得最彻底。

**OpenClaw 的"交响乐团"：** 异步事件驱动，父 Agent spawn 子 Agent 后立即返回。结果通过 announce 推送回来。Steer 机制允许运行中引导（2 秒/次，4000 字符）。查询进度不会误杀子 Agent。最多 8 路并发，嵌套深度可配。

**Hermes 的"总承包商"：** 同步阻塞，父 Agent 等所有子 Agent 完成。子 Agent 的中间推理不污染父 Agent 上下文。工具剥夺——子 Agent 不能再委派、不能访问记忆、不能与用户交互。最多 3 路并发，深度限 1 层。

| 维度 | OpenClaw | Hermes |
|------|----------|--------|
| 协作模式 | 异步事件驱动 | 同步阻塞 |
| Steer（运行时引导） | ✅（2s/次） | ❌（查询进度会误杀子 Agent） |
| 超时控制 | ✅（默认 300s） | ❌（可无限运行） |
| Token 效率 | 较低（+12% 开销） | 极高（上下文零膨胀） |
| 隔离性 | 弱（子 Agent 保留更多能力） | 强（工具剥夺 + 独立上下文） |
| 外部 Agent | ✅ ACP 协议 | ❌ |

OpenClaw 更灵活，Hermes 更安全。哪个更好取决于你的子 Agent 需要"听指挥"还是"别添乱"。

## 记忆系统：策展 vs 堆叠

OpenClaw 的记忆是**文件即记忆**的扁平模式：MEMORY.md（截断 200 行，不衰减）+ 每日笔记（时间衰减，半衰期 30 天）。BM25 + 向量双路召回。简单直观，你可以直接打开 MEMORY.md 看 Agent 记住了什么。

Hermes 的记忆是**分层策展**：3,575 字符的始终在线记忆 → FTS5 按需检索 → Skills 渐进加载 → Honcho 用户建模。Agent 自己决定什么值得记——Periodic Nudge 强制策展。

**核心差异**：OpenClaw 的记忆对人透明（打开文件就能看），Hermes 的记忆对 Agent 高效（分层检索更省 Token）。

一个有趣的细节：OpenClaw 的压缩算法花了大量工程投入——分块摘要、三级兜底、5 分钟超时保护——本质上是在**弥补扁平记忆模式的局限**。Hermes 因为一开始就做了分层，压缩问题反而不那么严重。

## 三个判断

**判断一：融合是趋势，但分化不会消失。**

Hermes 已经内置了 `hermes claw migrate` 迁移命令，可以安装 ClawHub Skills，agentskills.io 标准设计目标就是跨平台可移植。两者会互相借鉴——OpenClaw 会加强学习能力，Hermes 会扩展连接性——但底层哲学差异（生态 vs 学习）不会消失。这就像 iOS 和 Android 都在趋同，但设计哲学永远不同。

**判断二：安全会成为 Agent 选择的第一筛选器。**

在 Agent 只是"聊天"时，安全是次要考虑。当 Agent 拥有文件系统权、命令执行权、网络访问权——特别是在企业场景——安全变成了一票否决的条件。12-20% 的恶意 Skill 率不是可以接受的代价。这个因素会持续有利于约束式设计。

**判断三：你需要回答的不是"哪个更好"，而是"你的 Agent 活多久"。**

如果你的 Agent 是临时的——用完即弃，每次从零开始——选 OpenClaw。连接广度在短期内提供最大效用。

如果你的 Agent 是长期的——持续运行，积累经验，为你的特定场景越用越好——选 Hermes。认知深度的复利需要时间才能兑现，但一旦兑现，这是无法购买的竞争优势。

如果你两个都需要——等一年。融合正在发生。
