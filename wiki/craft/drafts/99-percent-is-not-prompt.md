# Prompt 决定下限，Harness 决定上限

> Prompt 写得再好，Agent 的表现也有天花板。真正决定 Agent 能走多远的，是 Prompt 之外的那些代码。

---

## 一个反直觉的数字

Claude Code 有超过 512,000 行代码。

猜一下，其中 Prompt 占多少？把所有 System Prompt、Tool Description、Few-shot Examples 加起来——几千行，撑死了。

**剩下的 99%+ 是什么？**

是错误恢复逻辑、工具编排系统、上下文压缩算法、安全沙箱、状态管理、可观测性追踪、评估闭环……所有那些模型"看不到"但决定模型表现的代码。

这些代码有一个名字：**Harness**。

Prompt 给 Agent 一个还不错的起点——它决定了下限。但 Agent 最终能有多好，取决于 Harness 的工程质量——它决定了上限。

---

## Prompt、Context、Harness：三层嵌套

很多人把 Prompt Engineering、Context Engineering、Harness Engineering 当成三个平行的技术路线。不是的。它们是嵌套关系：

```
Prompt ⊂ Context ⊂ Harness
```

- **Prompt** 管"怎么措辞"——指令文本、角色设定、few-shot examples
- **Context** 管"模型看到什么"——不只是 Prompt，还包括记忆管理、检索信息、工具输出、输出格式约束
- **Harness** 管"系统怎么运行"——不只是 Context，还包括所有模型"看不到"的基础设施

一个类比：Prompt 像是给员工发的一封工作邮件，Context 像是员工桌上的所有参考资料，Harness 像是员工的整个工作环境——办公室、电脑、工具、流程规范、绩效考核体系、同事支持。

邮件写得再好，如果工作环境一塌糊涂，员工也做不好事情。

### 数据说话

Princeton 研究做了一个精确的对照实验：拿同一个 Claude Opus 4.5，不改模型、不改 Prompt，**只改 Harness 配置**——一个 Harness 得分 2%，另一个得分 12%。

**6 倍性能差距，100% 归因于环境设计，而不是模型能力。**

他们的另一个发现更有冲击力：仅改变 Harness 配置，整体 solve rate 提升了 64%。这 64% 不是来自更好的模型、更巧妙的 Prompt——纯粹是 Harness 层面的工程改进。

MIT 的统计则从另一个角度印证了这一点：约 95% 的大企业 AI 试点未能产出可度量回报。根源不是 Prompt 质量——是 Harness 层的架构缺口。

## 那 99% 的代码在干什么？

让我们打开 Harness 的黑箱，看看模型之外到底有什么。

Akshay Pachaar 把一个 Agent Harness 系统化拆解为 12 个组件：

| 层次 | 组件 | Prompt 能覆盖吗？ |
|------|------|:--:|
| **核心** | 编排循环（Orchestration Loop） | ❌ |
| **核心** | 工具集成（Tools Integration） | ❌ |
| **核心** | 记忆管理（Memory Management） | ❌ |
| **核心** | 上下文管理（Context Management） | 一小部分 |
| **核心** | 模型集成（Model Integration） | ❌ |
| **核心** | 人机协同（Human-in-the-Loop） | ❌ |
| **健壮性** | 错误处理（Error Handling） | ❌ |
| **健壮性** | 安全防护（Security & Guardrails） | ❌ |
| **健壮性** | 状态管理（State Management） | ❌ |
| **质量** | 可观测性（Observability） | ❌ |
| **质量** | 评估系统（Evaluation） | ❌ |
| **质量** | 部署扩容（Deployment & Scaling） | ❌ |

Prompt 直接相关的只有"上下文管理"中的一小部分。其余 11 个组件，全在 Prompt 的射程之外。

Vivek Trivedy 说得更直接：**"If you're not the model, you're the harness."** 模型之外的一切——就是 Harness。

下面挑几个最关键的组件展开。

### 上下文策展：信号越纯越好，不是越多越好

大多数人对上下文的直觉是"给模型更多信息 → 模型更聪明"。

现实恰好相反。

Chroma 的研究显示：当关键内容被放在上下文窗口的中间位置时，模型的准确召回率下降超过 30%。信息越多，关键信号越容易被淹没——这个现象叫 **Context Rot**（上下文衰减）。

Vercel 在构建 AI 产品时发现了一个反直觉的现象：他们**删掉了 80% 的工具**，效果反而更好。不是工具没用，而是工具描述占据了宝贵的上下文窗口，稀释了真正重要的信息。

这就是为什么 Context Engineering 不是"把所有信息塞进去"，而是策展**最小可行的高信号 token 集**。

OpenClaw 的实现完美体现了这个原则：系统有 23 个模块可以注入上下文，但不是全部加载——`full` 模式加载所有模块（主 Agent），`minimal` 模式只保留核心模块（子 Agent），`none` 模式极简到只有身份标识。Skill 信息默认不加载，只在任务需要时才注入名称和描述。

**按需加载，而非一次性填满。**

### 上下文压缩：怎么优雅地丢弃

当对话足够长，上下文窗口终会接近上限。这时你必须做出选择：丢什么？

最简单的策略是截断——删掉最早的消息。但旧消息里可能包含关键决策（"我们选了方案 A 因为……"），任务开头的目标定义会被丢掉（模型"忘了"自己在干什么），文件哈希、UUID 等不透明标识符一旦丢失就无法恢复。

OpenClaw 选了一个更复杂但更"聪明"的策略——**分块摘要**：

```
上下文接近上限
    ↓
把旧消息切成块（每块约占上下文 40%）
    ↓
逐块让模型生成摘要
    ↓
用摘要替换原文
```

关键参数背后都是工程权衡：40% 的块大小意味着 200K 上下文窗口大约切 2-3 块，每块 80K token——够大才能保持连贯性，又不至于粒度太粗一次丢掉太多信息。

更精妙的是**保留列表**——这些信息绝对不能被压缩丢弃：

- **活跃任务**：正在做什么（丢了就不知道目标了）
- **重要决策**：为什么选了方案 A（丢了可能重新做错误决策）
- **TODO 列表**：还有什么没做（丢了就遗漏了）
- **不透明标识符**：UUID、文件哈希（摘要后面目全非）
- **用户承诺**："我答应你明天做 X"（丢了就失信了）

**知道什么不该丢，比知道什么该留更重要。**

这就是 Harness 在干的事情——Prompt 写不了压缩算法，但压缩算法决定了模型还记得什么。

### 记忆系统：让 Agent 不再每天失忆

没有记忆的 Agent 就像每天失忆的员工：每次上班都要重新自我介绍、重新学习流程、重新犯同样的错误。

但 Agent Memory 不是 RAG。RAG 检索外部文档——相当于查百科全书；Agent Memory 积累自身运行经验——相当于翻自己的工作日志。

| 维度 | RAG | Agent Memory |
|------|-----|--------------|
| 数据来源 | 外部文档（预先准备） | Agent 自身的运行轨迹 |
| 更新时机 | 离线索引 | 实时 / 会话结束时 |
| 内容性质 | 事实性知识 | 经验性知识 |
| 淘汰机制 | 通常没有 | 主动策展（有限空间强制淘汰） |

记忆系统的核心设计问题不是"怎么存"——往数据库写条记录谁都会。难的是**策展**：空间有限时，保留什么、丢弃什么？

OpenClaw 的设计很微妙：MEMORY.md（长期核心事实）不衰减，由人或 Agent 主动维护；每日笔记用时间衰减公式自动降权——半衰期 30 天，90 天后只剩 1/8 的权重。两者配合，模拟了人类记忆"核心事实 + 渐忘细节"的结构。

这些全是代码，没有一行是 Prompt。

### 安全防护：三层沙箱

Agent 能操作文件系统、能执行命令、能访问网络。如果你只靠 Prompt 说"请不要做危险操作"——你是在跟一个不完全可控的系统做口头约定。

OpenClaw 的做法是三层纵深防御：

1. **文件系统沙箱**：限制可读写的目录范围
2. **命令执行沙箱**：限制可执行的命令类型
3. **网络访问沙箱**：限制可访问的域名

再加上 **Human-in-the-Loop**：高风险操作（删除文件、执行系统命令）暂停等人确认。

安全不是靠 Prompt 实现的，是靠代码实现的。

## 一个常见的误解：Harness = Workflow？

到这里你可能会想："明白了，Harness 就是把 Agent 的步骤编排好——A 做完交给 B，B 做完交给 C。不就是高级版的 Workflow 吗？"

**不是。核心差异是主导权归属。**

| 维度 | Workflow | Harness |
|------|----------|---------|
| 路径 | 硬编码线性（A→B→C） | 动态软约束 |
| 模型角色 | 节点 / 执行者 | 自主规划 + 迭代 |
| 主导权 | 人 | AI（在约束内） |
| 异常处理 | 容易断裂 | 自动反思重试 |

用一个具体场景来理解——假设任务是"修复这个 Bug"：

**Workflow 方式**：读代码 → 定位问题 → 生成修复 → 运行测试 → 提交。每一步是预定义的，如果"运行测试"失败了，人提前写死了是回到"生成修复"还是报错退出。

**Harness 方式**：模型拿到任务 → 自己决定先读哪些文件 → 尝试修复 → 发现测试仍然失败 → 反思错误原因 → 换一个策略 → 再次测试 → 通过 → 提交。Harness 负责提供约束（只能修改指定目录、提交前必须通过测试、超时 5 分钟停止），但不规定执行顺序。

**Workflow 是给模型画好路线图，Harness 是给模型建好安全围栏。**

判断用哪个的关键问题：**这个任务的最佳执行路径，你能提前确定吗？**

- 能 → Workflow（比如 ETL 管线，步骤是确定的）
- 不能 → Harness（比如修复未知 Bug，你不知道需要读哪些文件、尝试几次）

而且有一个趋势：**模型能力越强，Harness 越优于 Workflow。** 当模型还不够聪明时，人类编排路径确实更可靠。但随着模型的推理和工具使用能力提升，让模型自主规划通常比人类预设路径效果更好。Harness 的适用范围在扩大，Workflow 的适用范围在缩小。

## 那 Prompt 不重要吗？

当然重要。Prompt 是起点——它定义了 Agent 的身份、目标、约束。一个烂 Prompt 会让其他所有组件的努力打折扣。

但 Prompt 的收益是递减的。

你把 Prompt 从"还行"调到"很好"，效果可能从 50 分提到 65 分。然后你花同样的时间把 Prompt 从"很好"调到"完美"——可能只从 65 分提到 67 分。

而同样的时间投入到 Harness 层——加一个错误恢复机制、优化上下文压缩策略、建立记忆系统——可能直接从 67 分跳到 85 分。

现在你能理解标题说的了：**Prompt 把下限兜住，Harness 把上限抬起来。** 两者缺一不可，但投资回报率完全不在一个量级。

Harrison Chase 的判断是：**未来 Agent 系统的核心竞争力不在模型，不在 Prompt，而在 Harness 捕获的轨迹。** 模型可能被下一代替换，Prompt 可以随时重写，但 Harness 积累的经验——哪些工具组合有效、哪些错误模式常见、什么样的上下文策略对特定任务最优——这些不会随模型更新而丢失。

这就是为什么 Phil Schmid 说："竞争优势不再是 prompt，而是你的 Harness 捕获的轨迹。"

## 从认识到行动

如果你正在构建 Agent 系统，以下是一个务实的优先级排序：

**第一步：先让 Prompt 过及格线**

花几天时间，把 System Prompt 写清楚——身份、目标、约束、输出格式。这是最低成本、最快见效的投入。但不要追求"完美 Prompt"，过了及格线就往下走。

**第二步：投资 Context Engineering**

想清楚模型需要看到什么信息，用什么策略加载：
- 哪些信息始终在（System Instructions）？
- 哪些按需加载（Skills、工具描述）？
- 上下文满了怎么办（压缩 vs 截断）？

记住：信号越纯越好，不是越多越好。

**第三步：搭建 Harness**

从最影响用户体验的组件开始：
1. **错误恢复**：模型出错时不要崩溃，自动重试或优雅降级
2. **安全防护**：限制 Agent 能做什么，高风险操作要人确认
3. **记忆系统**：让 Agent 的经验可以跨会话积累
4. **可观测性**：你得知道 Agent 在干什么、干得怎么样

**第四步：建立评估闭环**

有了前三步，Agent 能跑了。但怎么知道跑得好不好？怎么持续变好？

这就需要评估系统——收集 trace、度量质量、识别失败模式。最终可以走向 Meta-Harness：让编码 Agent 分析执行 Agent 的 trace，自动建议 Harness 改进。

## 结语

回到开头的数字：Claude Code 512,000 行代码，Prompt 只有几千行。

这不是巧合，不是 Anthropic 的过度工程。这是 Agent 系统的本质——**模型只是引擎，Harness 才是整辆车。**

下次有人问你"怎么让 Agent 更好"，不要条件反射地打开 Prompt 编辑器。先问自己：

> 这个问题，是 Prompt 能解决的，还是需要在 Harness 层面解决的？

99% 的情况下，答案是后者。
