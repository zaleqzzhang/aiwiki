# 视角：约束式设计 vs 无界式设计

## 核心问题

Agent 系统应该预设严格的资源边界（约束式），还是提供尽可能多的资源然后依赖智能管理（无界式）？

## 视角 A：约束式设计（Hermes 哲学）

**立场**：有限空间 → 强制策展 → 更高质量。约束不是限制，是创新的催化剂。

**核心体现**：

| 设计选择 | 约束 | 效果 |
|---------|------|------|
| 记忆上限 3,575 字符 | 强制策展 | 只保留最重要的信息 |
| 子 Agent 深度 = 2 | 防止递归 | 可预测的执行路径 |
| 子 Agent 工具剥夺 | 缩小行为空间 | 更高的可控性 |
| Skill 渐进式加载 | 只加载索引 | Token 效率最优 |
| 同步阻塞 | 禁止并行交叉 | 上下文零污染 |

**深层逻辑**：

约束迫使系统在"有限预算"内做出最优分配。就像创业公司资金有限所以每一分钱都花在刀刃上，Hermes 的 Token 预算有限所以每一个注入的信息都必须是"最值得注入的"。

Hermes 的 3,575 字符限制直接催生了四层记忆架构——如果一开始就给无限空间，可能就不会有这个精巧的分层设计。这和微服务中 bounded context 的逻辑一致：不是因为"装不下"才拆分，而是**拆分本身带来了更清晰的职责边界**。

**风险**：

- 信息丢失：策展失误可能导致重要信息被淘汰
- 能力天花板：深度限制、工具剥夺在复杂任务中可能不够
- 扩展性：约束参数（3,575、MAX_DEPTH=2）可能随需求增长需要反复调整

## 视角 B：无界式设计（MemOS 哲学）

**立场**：先全量保存，再用智能机制管理。不应该让人为的约束导致信息丢失。

**核心体现**：

| 设计选择 | 策略 | 效果 |
|---------|------|------|
| 35,000+ 条记忆 | 无上限存储 | 100% 信息保留 |
| 65,524 条关系边 | 记忆图谱 | 深层关联发现 |
| Hook 100% 捕获 | 系统强制 | 无遗漏 |
| 三层去重 | 智能过滤 | 信噪比控制 |
| 混合检索 | BM25 + 向量 + RRF | 多维度召回 |

**深层逻辑**：

无界式设计的赌注是：**智能管理的成本会随技术进步而下降，但丢失的信息永远找不回来。**

今天的 Token 成本可能让 35,000 条记忆的检索成本较高（$200-300/月），但随着模型推理成本下降，这个数字会变小。而如果你在早期因为约束丢掉了一条关键经验，后面要付出的代价可能更大。

MemOS 的哲学是："先存下来，总有一天你会需要它。" 这和数据工程中"先存 raw data，再按需 ETL"的思路一致。

**风险**：

- 成本高：存储、检索、去重都需要持续投入
- 复杂度高：三层去重、混合检索、质量评分——任何一环出问题都影响系统质量
- 维护负担：35,000 条记忆的数据质量治理不是一劳永逸的

## 更大的视角：这是工程哲学的永恒命题

约束式 vs 无界式不只是 Agent 架构的问题，它出现在几乎所有系统设计中：

- **数据库设计**：强 schema（SQL）vs 灵活 schema（NoSQL）
- **编程语言**：强类型（Rust/Go）vs 动态类型（Python/JS）
- **组织管理**：严格流程（瀑布式）vs 灵活流程（敏捷）
- **投资策略**：分散约束（止损线、仓位限制）vs 集中押注

每对选择都有各自的"甜区"——关键是认清你的场景更像哪一端。

**启发式判断**：

- 资源紧张、场景确定、需要可预测性 → 约束式
- 资源充裕、场景多变、需要发现性 → 无界式
- 不确定 → 从约束式开始（成本低），随需求增长逐步放开
