# 认知缺口：Harness ≠ Workflow

## 读者"以为懂"的认知

"Harness 就是一种高级 Workflow 嘛——把模型的调用步骤编排好，A 做完交给 B，B 做完交给 C。不就是 LangChain 那套 Chain 的思路？"

## 实际的关键区别

### 核心差异：主导权归属

| 维度 | Workflow | Harness |
|------|----------|---------|
| **路径** | 硬编码线性（A→B→C） | 动态软约束 |
| **模型角色** | 节点/执行者 | 自主规划+迭代 |
| **主导权** | 人 | AI（在约束内） |
| **灵活性** | 低 | 高 |
| **异常处理** | 容易断裂 | 自动反思重试 |

**Workflow 的隐喻是流水线**：人设计好每一步（提取信息 → 分类 → 生成回复 → 审核），模型只是流水线上的工人——在每个节点被调用一次，完成预设任务，没有自主权。路径是确定的，异常发生时流水线断裂。

**Harness 的隐喻是马具**：不是限制马怎么走，而是让马的力量可以被使用。模型保留自主规划和迭代的权利——它可以决定先做什么后做什么，遇到问题可以反思重试，甚至可以改变策略。Harness 提供的是约束边界（安全沙箱、权限控制、超时保护），而不是执行路径。

### 一个具体对比

假设任务是"修复这个 Bug"：

**Workflow 方式**：
```
读代码 → 定位问题 → 生成修复 → 运行测试 → 提交
```
每一步是预定义的，模型在每个节点被调用一次。如果"运行测试"失败了——回到"生成修复"重来？还是报错退出？这个决策是人提前写死的。

**Harness 方式**：
```
模型拿到任务 → 自己决定先读哪些文件 → 尝试修复 → 
发现测试仍然失败 → 反思错误原因 → 换一个修复策略 → 
再次测试 → 通过 → 提交
```
模型自主规划路径，Harness 负责提供约束（只能修改指定目录、提交前必须通过测试、超时 5 分钟自动停止），但不规定执行顺序。

### 飞樰从 OpenClaw 源码中的辨析

飞樰在分析 OpenClaw 源码时给出了最清晰的定义：

- **Prompt = What & How**（做什么、怎么做）
- **Context = How Better**（怎么做得更好）
- **Harness = How Controlled**（怎么可控地做）

Workflow 本质上还停留在"What & How"层面——它规定了做什么和怎么做（执行路径）。Harness 跳到了"How Controlled"层面——它不规定路径，但确保无论模型选择什么路径，结果都是可控的。

## 为什么这个误解常见

1. **历史惯性**：大多数人接触 AI 应用编排是从 LangChain 的 Chain/Agent 开始的，Chain 就是典型的 Workflow 思维。虽然 LangChain 后来加了 Agent 模式，但"链式调用"的心智模型已经根深蒂固。

2. **Workflow 确实更容易理解**：线性路径直觉上更可控——"我知道每一步会发生什么"。Harness 的"动态软约束"听起来更抽象、更不可预测。

3. **两者在简单场景中看起来一样**：如果任务足够简单（比如"翻译这段文字"），Workflow 和 Harness 的行为几乎没有区别。只有在复杂、需要迭代的场景中，差异才会显现。

4. **模型能力的分水岭**：模型能力越强，Harness 越优于 Workflow。当模型还不够聪明时，人类编排路径（Workflow）确实更可靠。但随着模型进步——特别是推理能力和工具使用能力的提升——让模型自主规划通常比人类预设路径效果更好。

## 正确的心智模型

**Workflow 是给模型画好路线图，Harness 是给模型建好安全围栏。**

- Workflow 假设：人比模型更知道该怎么做 → 所以人设计路径，模型执行
- Harness 假设：模型可能比人更知道该怎么做 → 所以模型自主规划，人设计约束

判断用哪个的关键问题：**这个任务的最佳执行路径，你能提前知道吗？**

- 能 → Workflow（比如数据处理管线：ETL 的步骤是确定的）
- 不能 → Harness（比如修复未知 Bug：你不知道需要读哪些文件、尝试几次）

随着模型能力提升，越来越多原本"能提前知道路径"的任务，实际上"让模型自己找路径"效果更好。**Harness 的适用范围在扩大，Workflow 的适用范围在缩小。**

## 写文章时可以这样切入

> "很多团队花了大量时间把 AI Agent 的每一步都编排好——先做这个、再做那个、失败了跳到这里。结果发现：模型经常比你预设的路径更聪明。它想往左走，你的 Workflow 硬让它往右——不是它不行，是你的编排限制了它。这就是 Workflow 和 Harness 的本质区别：一个是替模型做决定，一个是让模型做决定。"
