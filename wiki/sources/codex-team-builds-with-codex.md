# Codex 团队如何用自己的产品构建产品

**类型**: article
**日期**: 2026-04-06
**来源**: https://baoyu.io/blog/2026-04-06/how-openais-codex-team-builds-with-codex
**作者**: [[entities/baoyu|宝玉]]（翻译）
**标签**: #codex #openai #pm #workflow

## 摘要

OpenAI Codex 团队的产品规格文档只有 10 个要点——不是每个功能，而是整个产品的 spec。设计师写的代码量超过了六个月前工程师写的。50 到 100 人的团队，直到最近才有了第二个产品经理。Codex 产品负责人详解：10 个要点的 Spec、不做中期路线图、PM 从"领导者"变成"填空人"。

## 关键要点

### 10 个要点就够了

- Codex 团队几乎不写产品规格文档
- 只有涉及多人协调或重大决策时才写，往往只有 10 个要点
- **核心原则**："让离金属最近的人做尽可能多的决策"
- 一个人用 Codex 能处理的范围比过去大很多

### PM 用 AI 做"思维探索"

**三种模式**：

1. **简单改动直接上手**
   - 用 Codex 生成代码、测试、提交 PR
   - 直接提 PR 往往比找人沟通更快

2. **中等复杂度的改动**
   - 让 Codex 先做一个实现计划

3. **模糊想法的探索**
   - 在 Codex 里和模型对话，让它去代码库里探索、提问、生成方案
   - 不会真的使用这些方案（改动太复杂），但通过这个过程对问题有了更好的理解
   - 把这份理解（而非计划本身）分享给工程师

> "我不是在写代码，我是在建立心智模型。"

### 设计师写的代码超过工程师

- Codex 团队的设计师现在写的代码量，超过了六个月前一个工程师写的代码量
- 关键问题不再是"你能不能生成代码"，而是"你选择做的事情对不对"以及"你怎么保证质量"
- 凭感觉编程可以用来说"整个 app 是 AI 写的"，但 Codex 团队不是这么运作的

### 只做近期和远期规划，永远不做中期

**近期（8 周以内）**：
- 最多 8 周
- 具体的目标，团队能够围绕它集中发力
- OpenAI 擅长的短周期团队冲刺

**远期（方向感）**：
- 一年以后会有更聪明的模型
- 无限多个模型同时独立工作、自己验证结果、自己部署和监控代码
- 用户可能根本不需要主动输入提示词

**中间那段？**
- 产品路线图基本不存在
- 有的是长期方向 + 能朝那个方向推进的具体项目

> "在 OpenAI，你要么做近期规划，要么做远期规划，但永远不要做中期规划。"

### Codex App 的诞生

**关键转折点**：2025 年 12 月 GPT-5.2 Codex 发布
- 模型能力跨过了"可以可靠长时间独立工作"的门槛
- 用户开始用 tmux 同时运行大量 Codex 会话
- 社交媒体上流传：Peter Steinberger 在三块显示器上同时开了约 18 个终端窗口

**问题**：只有顶尖 1% 的工程师会用终端并行的方式工作
**方案**：做一个桌面应用，让多 Agent 协作变得自然

**技术架构**：Codex App、IDE 扩展、CLI 底层共用同一套开源 Rust 核心（Codex harness）

### 让最前沿的用户拉你进未来

- 先为 power user 做可配置性，再为普通用户做简化
- 最前沿的用户在和团队一起探索未来
- 子 Agent（sub-agents）概念：Codex 已在底层支持，用户可以自己启用、实验
- 核心交互做到极简，通过分层让 power user 自己去发现更深的功能

> "我们最前沿的用户和我们一起住在未来，还把我们往未来拽。"

### 50-100 人团队的"海盗船"式运作

**执行模式**：
- 大量使用 Codex 了解 Slack 反馈
- 用 Codex 总结反馈并发到 Linear
- 用 Codex 理解代码库状态
- 直接用 Codex 提交代码改动

**协调模式**：
- 花更多时间思考和沟通
- 用 Codex 更多是做文字工作

**团队特点**：
- 刻意保持低摩擦运作
- "有意识地像一支海盗船"
- 跨职能协调极少
- 直到最近才有了第二个 PM
- 大部分人都是工程师，汇报结构极简

### PM 不是领导岗位，是填空岗位

> "我其实不认为 PM 是一个好的领导岗位。我觉得它是一个填补空缺的岗位。"

- AI 工具让每个人都能做一部分别人的工作
- Scott Belsky 的"人才栈压缩"（collapsing the talent stack）正在发生

> "做任何事情所需的人越少，那件事就做得越好。每个决策都更纯粹。"

**两个细分**：
- 如果你一直想做工程师，只是因为管理能力不错才做了 PM → 可以直接转工程师
- 如果你真正感兴趣的是花大量时间和用户在一起、理解市场走向，而且团队足够大 → PM 可能还有存在的空间

> **每个问题都需要一个负责的人，但这个人不一定得是 PM。**

### 招聘标准：能动性（Agency）

- 没有递增难度的任务列表，就是"欢迎，自己找事做"
- 优先看对方发来的链接和想法
- 完全不知道团队成员是什么学校毕业的

> "我很高兴我们生活在一个那些愚蠢的证书不再重要的世界里。管它什么学校呢。给我看你做了什么。"
