# 飞书 CLI 开源了，为什么 AI Agent 时代，大家都在做命令行工具？

**类型**: article
**日期**: 2026-03-28
**来源**: https://baoyu.io/blog/2026-03-28/lark-cli-ai-agents
**作者**: [[entities/baoyu|宝玉]]
**标签**: #cli #mcp #skills #lark

## 摘要

飞书开源了 lark-cli，能让 AI Agent 直接操作飞书：发消息、查日历、写文档、建多维表格、发邮件、管任务。Google 也开源了 gws，让 AI Agent 操作 Google Workspace。2026 年，所有想接入 AI Agent 的产品都在做 CLI。为什么 CLI 在 AI Agent 时代重新火起来了？

## 关键要点

### CLI 为什么适合 AI Agent

**1. CLI 是自描述的**
- AI 碰到陌生的 CLI，敲一下 `--help` 就知道有哪些能力、怎么用、参数怎么填
- API 不行，AI 得先拿到文档、弄清端点、搞懂认证方式
- **CLI 自带说明书，AI 拿来就能用**

**2. CLI 天然是用文本交互的**
- 输入是文字，输出也是文字
- AI 最擅长处理的就是文字
- GUI 需要截图、视觉模型识别、模拟鼠标点击，一行命令能搞定的事拆成四步

### CLI vs MCP vs Skills

```
CLI    → 实际干活的工具（手）
MCP    → 另一种手，提前注册工具清单
Skills → 给 Agent 看的说明书（肌肉记忆）
```

**CLI 的优势**：
- AI 需要的时候自己去终端敲命令，用完就走，不占上下文
- 可以靠管道和参数组合出没预设过的操作

**MCP 的优势**：
- 在不支持命令行的环境（Cursor、Claude 桌面端）是唯一选择
- 工具清单常驻上下文窗口

**Skills 的作用**：
- 告诉 Agent 这个 CLI 有哪些命令、什么场景该用什么参数、出错了怎么处理
- 有了 Skills 文件，Agent 一上来就知道该怎么操作，成功率高得多

### 怎么给 AI 写好一个 CLI

**1. Help 文本是最重要的文档**
- AI 碰到不认识的 CLI，第一件事就是运行 `--help`
- 要写清楚每个参数干什么、什么时候用、有什么默认值
- 飞书 CLI 还有 `schema` 命令，查询任何 API 方法的参数、请求体、响应结构

**2. 支持 dry-run，为 AI 设计的安全网**
- AI 会自己做决策，有时候理解错了意图
- `--dry-run` 相当于"预览"机制
- Google 的 gws 技能文件写死规则：对所有写入和删除操作，必须先 dry-run

**3. 错误信息要能指导下一步操作**
- 人看到 `Permission denied` 会自己查文档
- AI 看到 `Permission denied` 就卡住了
- 飞书 CLI：告诉 AI 缺了什么权限，顺便把申请权限的命令也给出来

**4. 返回结构化数据，控制好输出量**
- 对 AI Agent 来说 JSON 更可靠
- AI 的上下文窗口有限，如果返回一万行日志，上下文就炸了
- 提供分页参数和过滤参数

> **让 Agent 动手之前，先让它 dry run 一遍。**

### CLI 的回归

```
过去四十年：CLI → GUI，从文字到图标，对人越来越友好
AI Agent 时代：方向反过来了，软件的用户变成了 AI Agent
```

> CLI 这个为文字世界设计的接口，恰好是 AI 最顺手的工具。

### 权限问题

- 不给权限，什么都做不了
- 权限太高，又怕 Agent 理解错意图干出不可逆的事
- dry-run 能兜住一部分风险
- 真正要让 Agent 在企业里大规模跑起来，权限体系、审计追踪、人机协作的边界，都还在摸索中
