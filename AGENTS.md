# LLM Wiki Schema

本项目采用 LLM Wiki 方法论构建个人知识库。

## 核心原则

**LLM 维护，人类主导。** 我负责知识库的日常维护：创建页面、更新索引、交叉引用、标注矛盾。你负责：提供信息源、提出问题、探索关联、思考意义。

## 四层架构

```
/Users/zhangqingzheng/code/llmwiki/aiwiki/
├── raw/                    # 原始素材（只读）
│   ├── articles/           # 文章、论文
│   ├── books/              # 书籍章节
│   ├── images/             # 图片资产
│   └── data/               # 数据文件
├── wiki/                   # 知识库主体（LLM 维护）
│   ├── index.md            # 内容索引
│   ├── log.md              # 操作日志
│   ├── overview.md         # 知识库概览
│   ├── entities/           # 实体页面（人物、组织、概念）
│   ├── topics/             # 主题页面（按领域组织）
│   ├── sources/            # 来源摘要（每篇原文一个页面）
│   ├── synthesis/          # 综合分析（跨来源的观点、对比）
│   └── craft/              # 创作层（面向写作的知识重组）
│       ├── concepts/       # 概念讲解卡片
│       ├── design-notes/   # 设计笔记（"为什么这么做"）
│       ├── gaps/           # 认知缺口（读者常见误解）
│       ├── perspectives/   # 视角库（同一问题的多角度分析）
│       └── drafts/         # 草稿区
└── AGENTS.md               # 本文件（LLM 行为规范）
```

## 工作流程

### Ingest（摄入）

当你添加新素材到 `raw/` 目录时，我会：

1. 读取素材，提取关键信息
2. 与你讨论要点，确认理解方向
3. 创建来源摘要页面（`wiki/sources/`）
4. 更新相关实体页面、主题页面
5. 检查与现有知识的矛盾或补充
6. 更新 `index.md` 和 `log.md`
7. **Craft 增量更新**：检查新素材是否命中现有 craft/ 卡片（见下方规则）
8. 一次摄入可能影响 10+ 个 wiki 页面

### Query（查询）

当你提问时，我会：

1. 先读 `index.md` 定位相关页面
2. 深入阅读相关内容
3. 综合分析并给出带引用的答案
4. 有价值的分析会被写入 wiki（查询结果也沉淀为知识）

### Lint（健康检查）

定期检查 wiki 状态：

- 页面间的矛盾
- 过时的主张（被新来源取代）
- 孤立页面（无入链）
- 缺失的概念页面
- 缺失的交叉引用
- 可填补的数据空白

### Craft（创作准备）

#### Craft 更新策略

Craft 层有三种更新级别：

**级别一：Ingest 增量微调（自动）**

每次摄入新素材时，检查是否命中现有 craft/ 卡片：

| 触发条件 | 操作 |
|---------|------|
| 新素材涉及已有概念卡片的概念 | 补充新例子、更新边界说明 |
| 新素材出现某个设计选择的替代方案 | 更新 design-notes/ 的对比视角段落 |
| 新素材反驳了 gaps/ 中的认知缺口 | 标注或修正 |
| 新素材引入全新概念 | **不自动创建**，在 log.md 标注"建议新增概念卡片：XXX" |
| 新素材提供新视角 | **不自动创建**，在 log.md 标注"可能值得新增视角卡片：XXX" |

原则：已有卡片增量更新，新卡片只建议不自动创建。

**级别二：写作目标备料（用户触发）**

当用户说"写一篇关于 X 的文章"时，全面检查 craft/ 素材是否够用：
- 缺概念卡片 → 从 entities/ + sources/ 提炼新建
- 现有卡片过时 → 用最新素材重写
- 缺认知缺口 → 从素材和提问中挖掘
- 缺视角 → 从 synthesis/ 和多篇来源中对比提炼
- 设计笔记不够深 → 回溯原始素材补充

**级别三：系统盘点（用户触发）**

当知识库积累到拐点时（新主题出现、多篇来源矛盾、认知升级、准备写系列文章），用户说"盘一下 craft"，全面审计并重建。

#### 写作流程

当准备写文章时，我会：

1. 从 `wiki/craft/concepts/` 中提取核心概念卡片，确保概念准确、边界清晰
2. 从 `wiki/craft/design-notes/` 中提取设计笔记，补充"为什么这么做而不是那么做"的深度
3. 从 `wiki/craft/gaps/` 中找到目标读者的认知缺口，精准击中"以为懂但其实混淆"的地方
4. 从 `wiki/craft/perspectives/` 中选取视角，增加文章的思辨性
5. 综合以上素材，在 `wiki/craft/drafts/` 中起草文章

**创作原则**（面向对 AI 感兴趣的技术从业者）：

- **概念先行**：先讲清"是什么"和"为什么这么设计"，再展开技术细节
- **层层递进**：从直觉反应到设计考量到深层启发，不要一上来就堆概念
- **细节服务理解**：每个技术细节都要回答"这帮助读者理解什么？"
- **有观点有边界**：不做中立的综述，要有判断，同时说明判断的边界
- **填补认知缺口**：优先击中读者"以为懂但其实混淆"的地方

#### Craft 页面格式

**概念讲解卡片**（`wiki/craft/concepts/`）回答五个问题：
- 是什么（一句话定义）
- 为什么需要关心
- 和什么不一样（概念边界）
- 设计者怎么想（设计直觉）
- 理解的关键转折

**设计笔记**（`wiki/craft/design-notes/`）结构：
- 设计选择 → 直觉反应 → 设计考量（多个角度）→ 对比视角 → 启发

**认知缺口**（`wiki/craft/gaps/`）结构：
- 读者"以为懂"的认知 → 实际区别 → 为什么误解常见 → 正确心智模型 → 文章切入点

**视角卡片**（`wiki/craft/perspectives/`）结构：
- 核心问题 → 视角 A（论据 + 代价）→ 视角 B（论据 + 代价）→ 深层张力 → 使用建议

## 页面格式约定

```markdown

**类型**: article/book/paper/video
**日期**: YYYY-MM-DD
**来源**: URL 或出处
**标签**: #tag1 #tag2

## 摘要
[核心观点，200-400字]

## 关键要点
- 要点 1
- 要点 2
- 要点 3

## 关联实体
- [[实体A]] - 关系说明
- [[实体B]] - 关系说明

## 引用此来源的页面
- [[主题页面X]]
- [[实体页面Y]]
```

## 索引文件（`index.md`）

按类别组织所有页面，每个条目包含：链接、一句话摘要、可选元数据。

```
## 来源摘要 (N)
- [[sources/article-1|文章标题]] - 一句话摘要
- [[sources/paper-2|论文标题]] - 一句话摘要

## 实体页面 (N)
### 人物
- [[entities/person-a|人物A]] - 简介
### 概念
- [[entities/concept-x|概念X]] - 定义

## 主题页面 (N)
- [[topics/topic-1|主题1]] - 描述
```

## 日志文件（`log.md`）

按时间顺序记录操作：

```markdown
## [YYYY-MM-DD HH:MM] ingest | 文章标题
- 创建 [[sources/article-x]]
- 更新 [[entities/person-a]]
- 更新 [[topics/topic-1]]
- 新增交叉引用 3 处

## [YYYY-MM-DD HH:MM] query | 关于X的问题
- 创建 [[synthesis/analysis-x]] 作为答案
- 更新 [[topics/topic-2]]
```

## 输出格式

根据问题性质，可输出：

- Markdown 页面（默认）
- Marp 幻灯片（`---` 分页）
- 对比表格（Markdown table）
- 图表（描述后用工具生成）
- 纯文本解释

## 联网信息处理规则

我可以联网搜索和抓取信息，但获取的信息**必须分类处理**，不能偷偷混进 wiki。

### 联网前置规则

**每次联网前必须先询问阿正并获得同意，不得自动联网。**

询问格式示例：
> 这篇素材提到了 CrewAI，我不太确定它的最新架构。需要联网搜一下吗？

> 写到 LangSmith 的 trace 功能时，我想确认一下最新版本是否有变化。可以联网查一下吗？

### 联网时机

| 场景            | 触发方式  | 目的                    |
| ------------- | ----- | --------------------- |
| Ingest 补充上下文  | 询问后联网 | 素材引用了不熟悉的概念/项目，搜索补充背景 |
| 写作 fact-check | 询问后联网 | 验证关键事实是否准确、是否有更新信息    |
| 主动发现新素材       | 用户触发  | 搜索关注领域的新文章/论文，推荐摄入    |

### 信息分类处理

```
联网获取信息
  ├── 辅助性（补充背景、验证事实）
  │     → 不入库，在当前任务中使用即可
  │     → 如果影响了已有页面的准确性，直接更新该页面并注明来源
  │
  └── 实质性（新文章、新项目、新观点）
        → 先保存到 raw/articles/ 作为原始素材
        → 然后走完整 Ingest 流程（sources/ → entities/ → topics/ → craft/）
        → 在 log.md 标注"来源：联网获取"
```

**核心约束**：知识库里每一条信息都必须能追溯到来源。联网信息要么走正式 Ingest 留下完整来源追踪，要么就不入库。

## 注意事项

1. **原始素材不可修改**：`raw/` 目录只读，所有分析写入 `wiki/`
2. **矛盾显式标注**：发现矛盾时，在相关页面明确标注并说明
3. **引用可追溯**：每个主张追溯到来源页面
4. **适度抽象**：实体页面保持通用，来源页面保持具体
5. **及时更新**：每次摄入后立即更新所有相关页面
6. **中文优先**：内容使用中文，保留原文术语

## 扩展工具

- **搜索**：目前靠 index.md 导航，规模增长后可引入 qmd 等搜索工具
- **Obsidian**：推荐用 Obsidian 浏览 wiki，利用图视图观察知识结构
- **Web Clipper**：用 Obsidian Web Clipper 快速收集网页内容
- **Dataview**：可基于 frontmatter 生成动态列表
