**CLAUDE.md** is a configuration file that provides [Claude Code](https://claude.ai/code) — Anthropic’s agentic coding tool — with persistent, project-specific instructions. It is automatically loaded before every conversation, giving the [agent](https://latentpatterns.com/glossary/agent) immediate context about the codebase’s architecture, build commands, testing procedures, and coding conventions without the developer needing to repeat them. The filename is case-sensitive: `CLAUDE.md` (uppercase `CLAUDE`, lowercase `.md`).

Claude Code reads instructions from multiple locations, organized in a **memory hierarchy** where more specific scopes override broader ones:

**Managed policy** files live in system directories (`/etc/claude-code/CLAUDE.md` on Linux, `/Library/Application Support/ClaudeCode/CLAUDE.md` on macOS) and are controlled by IT or DevOps — individual developers cannot override them. **Project memory** (`./CLAUDE.md` and `.claude/rules/*.md`) and **user memory** (`~/.claude/CLAUDE.md`) load automatically at session startup. **Local project memory** (`./CLAUDE.local.md`) is automatically added to `.gitignore`, making it the right place for personal preferences that should not be checked in. **Child directory** `CLAUDE.md` files load conditionally — only when Claude reads or edits files in that directory, keeping the [context window](https://latentpatterns.com/glossary/context-window) lean for focused tasks. Finally, **auto memory** is a directory where Claude itself saves learnings and patterns it discovers while working; only the first 200 lines are loaded per session. In [monorepos](https://latentpatterns.com/glossary/monorepo), this layered loading means each package can carry its own `CLAUDE.md` with package-specific build commands and conventions without inflating the base prompt.

Beyond flat `CLAUDE.md` files, Claude Code supports **scoped rules** in `.claude/rules/*.md`. Each rule file can include [YAML](https://latentpatterns.com/glossary/yaml) frontmatter with a `paths:` field that restricts when the rule activates — for example, a rule targeting `*.test.ts` files might enforce specific testing conventions, while a rule targeting `src/lib/server/**` might describe database access patterns. The `/init` slash command auto-generates a starter `CLAUDE.md` by analyzing the project’s structure, detecting the tech stack, and inferring common commands — though the output is a starting point that should be refined with the team’s actual architectural decisions and coding standards.

CLAUDE.md is a **vendor-specific** instruction format — it is read only by Claude Code. The cross-tool alternative is [AGENTS.md](https://latentpatterns.com/glossary/agents-md), an open standard governed by the Agentic AI Foundation under the Linux Foundation and supported by a growing set of coding tools (including OpenAI Codex, GitHub Copilot, and Cursor). Claude Code does not natively read `AGENTS.md` — it looks only for `CLAUDE.md`. Teams that want a single source of truth can bridge the gap with a **symlink**: the canonical instructions live in `AGENTS.md` (version-controlled), and a symlink at `CLAUDE.md → AGENTS.md` ensures Claude Code continues to find them at its expected path. Claude Code resolves symlinks transparently — reads and edits go through to the target file. The symlinked `CLAUDE.md` can be added to `.gitignore` so only `AGENTS.md` is tracked. This is the recommended approach for teams using multiple AI coding tools, as it eliminates content drift between duplicate instruction files.

The primary authoring trade-off is **conciseness versus completeness**. Every line of `CLAUDE.md` is injected into the agent’s prompt and competes for [context window](https://latentpatterns.com/glossary/context-window) tokens alongside conversation history, [tool calling](https://latentpatterns.com/glossary/tool-calling) definitions, and the model’s output. Vague instructions like “follow best practices” waste tokens without changing behavior; specific instructions like “use ES modules with named exports, never default exports” directly influence code generation. The best `CLAUDE.md` files read like a terse onboarding checklist for a senior engineer joining the team — commands to run, patterns to follow, mistakes to avoid, and nothing else.

Like any instruction file loaded by an [agent](https://latentpatterns.com/glossary/agent), `CLAUDE.md` introduces a trust surface. A malicious `CLAUDE.md` in a cloned repository can contain [prompt injection](https://latentpatterns.com/glossary/prompt-injection) payloads — instructions designed to override the agent’s safety guidelines or direct it to execute harmful [tool calls](https://latentpatterns.com/glossary/tool-calling). Teams should review `CLAUDE.md` files in third-party repositories with the same caution they apply to `Makefile`, `package.json` scripts, or CI configuration — any file that an automated system executes on trust.
